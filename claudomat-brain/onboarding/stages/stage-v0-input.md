# Stage v0 — Input: Receive Service / Product Description Documents

## Purpose
Receive founder-provided documents describing the service/product. Raw material for every subsequent onboarding stage. No analysis — acquisition, normalization, storage.

## Prerequisites
- Project scaffold from `claudomat init` exists (`CLAUDE.md`, `project.yaml`, `command-center/`, `design/`, `process/`, vendored `claudomat-brain/`).
- `claudomat-brain/onboarding/onboarding-loop.md` has been read.

## Actions

### Action 0 — HARD GATE: external-dependency installation must be complete

v0 refuses to proceed if any external dependency declared in `claudomat-brain/setup-tools/install.md` is missing. **Silent degradations downstream are forbidden.**

Run the gate (strict mode — WARNs are fatal):

```bash
claudomat doctor --strict
```

Exit 0 = proceed. Exit 1 = at least one external-dependency item missing → STOP and resolve per the loop below.

(The legacy `CLAUDOMAT_DOCTOR_STRICT=1 claudomat doctor` env-prefix form is still supported but discouraged inside hosted workers — Claude Code harness treats env-prefixed commands as compound multi-operation and may block them under non-interactive permission modes.)

Treat **every `[FAIL]` line as a hard-stop** and **every `[WARN]` line for an external-dependency item** as a hard-stop too. WARN markers exist so wave-loop hiccups don't crash the founder mid-wave; at v0 entry, they are blockers.

**On any FAIL or external-dependency WARN:**

1. STOP. Do NOT proceed to step 1.
2. Surface the missing items verbatim from `claudomat doctor`.
3. Cite the install.md phase that owns the missing item.
4. Offer one batched `AskUserQuestion` with options-and-custom:

   > "External dependencies missing — v0 cannot proceed until resolved. Pick:"
   >
   > 1. **Install all missing now** — I'll walk through each missing item invoking the install command from `claudomat-brain/setup-tools/install.md`. ~5–30 min.
   > 2. **Install one specific item** — tell me which; I'll run that install + re-verify; we loop until everything is `[OK]`.
   > 3. **You install manually** — you handle it; tell me when to re-run `claudomat doctor`.
   > 4. **Custom** — describe what you want to do.

5. After install: re-run `claudomat doctor`. If still any FAIL or external-dep WARN, repeat.
6. Only when `claudomat doctor` returns clean (zero FAILs, no external-dep WARNs — ignorable WARNs are: capability-sheet stale (auto-regens), test-accounts unfilled (v13 fills), onboarding-complete missing (this run will create it)), proceed to step 1.

**Conditional install items** — stack-conditional items (e.g., Stripe CLI only if v5 picks Stripe) are NOT gated at v0 because the stack isn't selected yet. Gated at the relevant later stage:

- v5 — re-runs `claudomat doctor` after stack selection; refuses to exit if any stack-conditional CLI / SDK is missing.
- v6 — verifies bespoke per-stack agents exist (or invokes agent-creator).
- v7 — verifies `/aidesigner` slash skill is wired.

This gate is the difference between *installing claudomat* (Phase 1–10 of install.md, founder + Claude collaboration outside the wave loop) and *running claudomat* (v0 onward, fully orchestrated). The former MUST be 100% complete before the latter starts.

### 1. Solicit input — options-and-custom

If no input documents are present in the conversation context already, fire `AskUserQuestion`:

> "How would you like to provide the product/service description? I'll work with whatever you give me — written briefs, decks, vision docs, competitor notes, persona research, prior project artifacts."
>
> 1. **Paste in chat** — drop the content directly, I'll capture verbatim.
> 2. **Link to source** — Google Docs / Notion / public URL / git repo. I'll fetch and normalize to local Markdown.
> 3. **Drop files** — put one or more files in `process/session/onboarding/docs-input/` and tell me you're done.
> 4. **Voice / stream-of-consciousness** — talk through the vision. I'll capture verbatim and structure minimally.
> 5. **Custom** — tell me how you want to deliver and I'll work with it.

If the founder picks **Custom**, capture the free-text response, restate as a concrete option, and confirm via `Confirm / Refine / Cancel` before proceeding.

### 2. Normalize and store

Per chosen option:

- **Paste** → write verbatim to `process/session/onboarding/docs-input/initial-brief.md` with YAML front-matter:
  ```yaml
  ---
  source: founder-paste
  received_at: <ISO-timestamp>
  word_count: <number>
  ---
  ```
- **Link** → fetch via WebFetch (or `gh repo clone` for git repos). Strip nav/chrome; preserve original wording. Write to `process/session/onboarding/docs-input/<kebab-case-title>.md` with `source: <url>` front-matter.
- **Files** → read each file in `docs-input/`; if non-Markdown, normalize to `.md` (preserving original prose). Add front-matter recording original filename + receipt timestamp.
- **Voice** → transcribe verbatim into `initial-brief.md` with `source: founder-voice` front-matter.
- **Custom** → apply founder-confirmed delivery mechanism; write to `docs-input/` with `source: custom-<descriptor>` front-matter.

Preserve the founder's original wording. Do NOT paraphrase, summarize, or "improve" prose. v1 does the structured extraction.

### 3. Sanity check — what + who

Confirm at least ONE of the following is present across the received docs:

- A one-line description of what the product does
- A named target user / market

If BOTH are absent, fire `AskUserQuestion`:

> "The docs you provided don't contain a product description or target market. I need at minimum: (a) what does this product do, and (b) who is it for. Pick one:"
>
> 1. **Quick one-liner pair** — I'll ask two short follow-ups: 'What does it do?' and 'Who is it for?'
> 2. **Expand the brief** — paste / link a fuller version of the docs.
> 3. **Custom** — tell me how you want to fill the gap.

Append resolved answers to `initial-brief.md`.

### 4. Optional snapshot commit

If received docs total >5 KB and could evolve during onboarding:

```bash
git add process/session/onboarding/docs-input/
git commit -m "chore(onboarding): v0 — input docs received"
```

Optional. v13 owns the final initial commit.

## Deliverable

- `process/session/onboarding/docs-input/*.md` — all received documents in original form, with `source:` front-matter.
- (If founder pasted / spoke) `process/session/onboarding/docs-input/initial-brief.md` populated.

## Exit criteria

- At least one document in `process/session/onboarding/docs-input/` with content length > 200 chars.
- Both "what does it do" and "who is it for" are answerable from the content.
