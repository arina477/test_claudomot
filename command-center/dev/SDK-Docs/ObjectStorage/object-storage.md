## Backend Decision Record

### Options evaluated (wave-19 M3 research — 2026-06-30)

| Option | Self-provisionable with APP_RAILWAY_TOKEN | Founder cred-ask | Fit |
|---|---|---|---|
| **Railway Buckets (chosen)** | Yes — `railway bucket create` via CLI; creds auto-generated by Railway | No — creds come from Railway dashboard, not an external account | Full S3-compatible API; presigned URLs; free egress; $0.015/GB-month; already in use for avatars |
| Self-hosted MinIO on Railway | Yes — deploy Docker container + Volume | No — access/secret keys self-generated | Valid fallback only; adds operational overhead (container lifecycle, Volume backup) vs. managed Railway Buckets |
| Cloudflare R2 | No | Yes — Cloudflare account ID + R2 access key + secret from Cloudflare console | Best external fallback; free egress; $0.015/GB-month; only needed if Railway Buckets infeasible |
| AWS S3 | No | Yes — IAM access key + secret from AWS console | Industry standard but over-engineered for this scale; egress costs |
| Backblaze B2 | No | Yes — Backblaze account + application key | Cheapest storage ($0.006/GB) but adds external account |

### Railway Buckets feasibility verdict

Railway Buckets is a first-class managed product on Railway as of 2026. Confirmed capabilities:
- **Provision:** `railway bucket create <name> --region <region>` — works against existing Railway project with `APP_RAILWAY_TOKEN`
- **S3-compatible API:** Full PutObject, GetObject, HeadObject, DeleteObject, ListObjects v2, CopyObject, Multipart uploads, Presigned URLs (up to 90 days)
- **Credential injection:** Railway auto-injects bucket credentials as env vars (`AWS_ENDPOINT_URL`, `AWS_ACCESS_KEY_ID`, `AWS_SECRET_ACCESS_KEY`, `BUCKET`) into connected services — or inject via `railway bucket credentials` CLI output
- **URL style:** Virtual-hosted-style (`https://<bucket-name>.storage.railway.app`) for buckets created after the change; older buckets may use path-style (`forcePathStyle: true`)
- **Public access:** No public buckets today; all access via presigned URLs or backend proxy. This is the correct model for attachment serving (auth check before URL generation)
- **Networking:** Public endpoint only (`https://storage.railway.app`); no private networking support yet
- **Cost:** $0.015/GB-month; all S3 API operations free; egress from bucket free (service egress for uploads is billed at standard rate)
- **Backed by:** Tigris on Railway's metal servers — real object storage, not block storage

**No founder cred-ask required.** Credentials are generated by Railway itself when the bucket resource is created; the deployer copies them from the Railway dashboard (or the CLI `credentials` command). No external service account is needed.

### Self-hosted MinIO comparison (why Railway Buckets wins)

MinIO on Railway is technically feasible (`minio/minio` Docker image + persistent Volume). However:
- Adds an additional Railway service to manage (healthchecks, restart policy, image updates)
- Volumes on Railway are block storage — object storage semantics layered on top (fine, but not native)
- No Railway-managed credential injection; secrets must be self-generated and wired manually
- No built-in durability guarantees beyond the Volume; Railway Volumes do offer PITR but this is additional configuration
- Railway Buckets (Tigris-backed) provides managed durability, encryption at rest, and S3 compatibility with zero operational overhead
- Railway Buckets costs the same per-GB as MinIO's storage costs (minus the service cost of running MinIO itself)

MinIO remains the correct fallback if Railway Buckets is ever deprecated or unavailable in a target region.

## Official API Surface

### Install

```bash
pnpm add @aws-sdk/client-s3 @aws-sdk/s3-request-presigner
```

Both packages are already in apps/api/package.json at ^3.1075.0.

### Public classes / functions

**@aws-sdk/client-s3:**
- `S3Client` — low-level client (recommended for bundling; bare-bones, no pagination helpers)
- `S3` — aggregated client (extends S3Client; adds all operations as methods)
- `PutObjectCommand` — upload an object
- `GetObjectCommand` — download an object / get its stream
- `HeadObjectCommand` — get metadata only (ContentLength, ContentType, ETag) without body
- `DeleteObjectCommand` — remove a single object
- `DeleteObjectsCommand` — batch delete (up to 1000 keys)
- `ListObjectsV2Command` — list objects in a bucket (paginated)
- `CopyObjectCommand` — server-side copy within or across buckets
- `CreateMultipartUploadCommand` / `UploadPartCommand` / `CompleteMultipartUploadCommand` / `AbortMultipartUploadCommand` — multipart upload for large files (not needed at ≤10MB limit)

**@aws-sdk/s3-request-presigner:**
- `getSignedUrl(client, command, options)` — returns a presigned URL string for any S3 command
  - `options.expiresIn` — TTL in seconds (default: 900; max for Railway Buckets: 7,776,000 i.e. 90 days)

### Constructor options (S3Client)

```typescript
import { S3Client } from '@aws-sdk/client-s3';

const s3 = new S3Client({
  // Required for non-AWS S3-compatible endpoints
  endpoint: process.env.AWS_ENDPOINT_URL,        // 'https://storage.railway.app'
  region: process.env.AWS_REGION ?? 'auto',       // 'auto' for Tigris / Railway Buckets

  credentials: {
    accessKeyId: process.env.AWS_ACCESS_KEY_ID!,
    secretAccessKey: process.env.AWS_SECRET_ACCESS_KEY!,
  },

  // Path-style vs. virtual-hosted-style:
  // Railway Buckets created after the URL-style migration use virtual-hosted style (default).
  // The Railway dashboard Credentials tab shows which style the bucket requires.
  // For buckets created before the migration, set forcePathStyle: true.
  // The existing FilesService uses forcePathStyle: true — review against bucket credentials tab.
  forcePathStyle: true,   // set to false for virtual-hosted buckets (newer Railway Buckets default)
});
```

**Critical:** `forcePathStyle` must match the URL style reported by the Railway dashboard Credentials tab for the specific bucket. Mismatch = 404/403 on all requests. Check the bucket's Credentials tab, not just the endpoint.

### Methods with signatures

All commands follow the same pattern: `client.send(new XxxCommand(input))`.

**PutObject (server-side upload):**
```typescript
import { PutObjectCommand } from '@aws-sdk/client-s3';

await s3.send(new PutObjectCommand({
  Bucket: process.env.STORAGE_BUCKET_NAME,
  Key: 'attachments/channel-123/uuid.pdf',
  Body: buffer,                 // Buffer | Readable | Uint8Array | string
  ContentType: 'application/pdf',
  ContentLength: buffer.length, // recommended for streams; prevents chunked encoding issues
}));
```

**HeadObject (existence + size check):**
```typescript
import { HeadObjectCommand } from '@aws-sdk/client-s3';

const head = await s3.send(new HeadObjectCommand({
  Bucket: process.env.STORAGE_BUCKET_NAME,
  Key: 'attachments/channel-123/uuid.pdf',
}));
// head.ContentLength → number of bytes
// head.ContentType  → MIME type set at upload time
// head.ETag         → MD5 hex quoted string
```

**GetObject (streaming download):**
```typescript
import { GetObjectCommand } from '@aws-sdk/client-s3';

const obj = await s3.send(new GetObjectCommand({
  Bucket: process.env.STORAGE_BUCKET_NAME,
  Key: 'attachments/channel-123/uuid.pdf',
}));
// obj.Body is a ReadableStream (Web Streams API) in Node 18+
// Pipe to response: obj.Body.pipe(res)
```

**DeleteObject:**
```typescript
import { DeleteObjectCommand } from '@aws-sdk/client-s3';

await s3.send(new DeleteObjectCommand({
  Bucket: process.env.STORAGE_BUCKET_NAME,
  Key: 'attachments/channel-123/uuid.pdf',
}));
// Does NOT throw if key does not exist — idempotent
```

**Presigned PUT (client uploads directly — bypass server):**
```typescript
import { PutObjectCommand } from '@aws-sdk/client-s3';
import { getSignedUrl } from '@aws-sdk/s3-request-presigner';

const key = `attachments/${channelId}/${randomUUID()}.${ext}`;

const command = new PutObjectCommand({
  Bucket: process.env.STORAGE_BUCKET_NAME,
  Key: key,
  ContentType: contentType,
  // ContentLength cannot be enforced in presigned-PUT. Use HeadObject at confirm time.
});

const uploadUrl = await getSignedUrl(s3, command, {
  expiresIn: 300,  // 5 minutes for the upload window
  // To enforce content-type on the PUT: add it to signableHeaders
  // signableHeaders: new Set(['content-type']),
});
// Return { uploadUrl, key } to client. Client PUTs binary to uploadUrl.
```

**Presigned GET (temporary download link):**
```typescript
import { GetObjectCommand } from '@aws-sdk/client-s3';
import { getSignedUrl } from '@aws-sdk/s3-request-presigner';

const command = new GetObjectCommand({
  Bucket: process.env.STORAGE_BUCKET_NAME,
  Key: 'attachments/channel-123/uuid.pdf',
});

const downloadUrl = await getSignedUrl(s3, command, {
  expiresIn: 3600,  // 1 hour for download URLs
});
// Return downloadUrl to authenticated client. No backend egress for the download itself.
```

**Presigned POST (upload with server-enforced constraints — preferred for file-size enforcement):**
```typescript
import { createPresignedPost } from '@aws-sdk/s3-presigned-post';
// Note: separate package — @aws-sdk/s3-presigned-post (not currently installed)

const { url, fields } = await createPresignedPost(s3, {
  Bucket: process.env.STORAGE_BUCKET_NAME,
  Key: key,
  Expires: 300,
  Conditions: [
    ['content-length-range', 1, 10 * 1024 * 1024],    // 10MB max — enforced by S3
    ['starts-with', '$Content-Type', 'image/'],         // content-type prefix
  ],
});
// Client POSTs multipart/form-data with fields + file to url.
// Advantage: size enforcement is server-side at upload time (no confirm-step HeadObject needed).
// Current project uses presigned-PUT + HeadObject confirm pattern — both are valid.
```

## Platform Compatibility

### Railway (deploy_platform)

**Deployment:** Railway Buckets is a first-class Railway resource. Create with:
```bash
railway bucket create studyhall-uploads --region iad   # or sjc / ams / sin
```

Available regions: `sjc` (US West), `iad` (US East), `ams` (EU West), `sin` (Asia Pacific).

**Credential injection — two options:**

Option A — Manual variable references (current approach):
Set the following on the API service in Railway:
```
AWS_ENDPOINT_URL     → (bucket reference) ${{studyhall-uploads.ENDPOINT}}
AWS_ACCESS_KEY_ID    → (bucket reference) ${{studyhall-uploads.ACCESS_KEY_ID}}
AWS_SECRET_ACCESS_KEY → (bucket reference) ${{studyhall-uploads.SECRET_ACCESS_KEY}}
AWS_REGION           → (bucket reference) ${{studyhall-uploads.REGION}}
STORAGE_BUCKET_NAME  → (bucket reference) ${{studyhall-uploads.BUCKET}}
```

Option B — Auto-inject (Railway dashboard Credentials tab → AWS SDK preset):
Railway injects all `AWS_*` env vars automatically when you select the AWS SDK preset in the bucket's Credentials tab. This sets exactly the vars the AWS SDK reads by default (`AWS_ENDPOINT_URL`, `AWS_ACCESS_KEY_ID`, `AWS_SECRET_ACCESS_KEY`, `AWS_REGION`).

**URL style:** Railway Buckets default to virtual-hosted style (`https://<bucket-name-hash>.storage.railway.app`) for buckets created after the June 2025 URL style migration. The existing FilesService sets `forcePathStyle: true` — this must be corrected to `forcePathStyle: false` (or removed, as false is the default) for buckets created after the migration. The bucket's Credentials tab in Railway dashboard is definitive.

**Private networking:** Not available for buckets — public endpoint only (`https://storage.railway.app`). Service egress applies for uploads from Railway services to buckets.

**Per-environment isolation:** Each Railway environment (production, staging, PR environments) gets its own separate bucket instance with isolated credentials. No cross-environment data bleed.

### NestJS (backend runtime)

**S3Client instantiation:** Lazy initialization in a NestJS `@Injectable()` service is the correct pattern. The existing FilesService uses a private `_s3Client` initialized on first use — this prevents boot failures when storage env vars are absent (graceful degradation: 503 STORAGE_NOT_CONFIGURED).

**Node.js runtime:** AWS SDK v3 is fully Node.js 22 compatible. `GetObjectCommand` returns `Body` as `Readable` (Node.js stream) in Node runtime — use `.pipe(response)` for streaming downloads. In edge/browser runtimes it returns `ReadableStream` (Web Streams API).

**NestJS DI pattern for S3Client:**
```typescript
// files.module.ts
@Module({
  providers: [FilesService, FilesController],
  exports: [FilesService],
})
export class FilesModule {}

// FilesService creates S3Client lazily — no NestJS DI for the S3Client itself,
// which is correct given the graceful-degradation pattern (null client when unconfigured).
```

### Vite + React (frontend runtime)

The AWS SDK v3 is browser-compatible, but the S3 client should NOT be used directly in the browser for this project. The correct pattern is:
1. Frontend requests a presigned URL from the NestJS API
2. Frontend PUTs binary directly to the presigned URL (`fetch(uploadUrl, { method: 'PUT', body: file }`)
3. This avoids exposing AWS credentials in the browser

The `@aws-sdk/client-s3` package should remain in `apps/api` only.

## Known Gotchas

### 1. forcePathStyle must match bucket URL style

Railway Buckets created before June 2025 use path-style URLs (`https://storage.railway.app/<bucket>/<key>`). Buckets created after use virtual-hosted style (`https://<bucket-hash>.storage.railway.app/<key>`). The S3 SDK default is virtual-hosted. Setting `forcePathStyle: true` on a virtual-hosted bucket causes 301 redirects or DNS failures. **Always check the Railway Credentials tab for the bucket's declared URL style.**

The existing FilesService hard-codes `forcePathStyle: true`. This was correct for the initial bucket. For new buckets created for wave-19 attachments, verify the URL style before assuming path-style.

### 2. presigned-PUT cannot enforce ContentLengthRange

`PutObjectCommand` presigned URLs cannot carry a `content-length-range` condition — that is a presigned-POST-only feature (`createPresignedPost` from `@aws-sdk/s3-presigned-post`). The existing avatar upload pattern correctly handles this by issuing a `HeadObjectCommand` at confirm time to enforce the 2MB cap server-side. Apply the same pattern for attachment uploads (10MB cap at confirm time via HeadObject).

Alternative: switch to `createPresignedPost` (presigned-POST) which supports `content-length-range` natively — the client then POSTs multipart/form-data instead of a raw PUT. Both patterns are valid; presigned-PUT is simpler for the client; presigned-POST enforces size server-side at upload time without a confirm step.

### 3. contentType must be signed in presigned-PUT for strict enforcement

If `contentType` is included in the `PutObjectCommand` at presign time and you want to enforce that the client sends that exact content-type on the PUT, pass it to `signableHeaders`:
```typescript
await getSignedUrl(s3, command, {
  expiresIn: 300,
  signableHeaders: new Set(['content-type']),
});
```
Without `signableHeaders`, the client can override `Content-Type` on the PUT. For a content-type allowlist (image/png, image/jpeg, image/webp), enforcing via `signableHeaders` is the correct pattern.

### 4. `BUCKET` vs `STORAGE_BUCKET_NAME` env var name mismatch

Railway's `railway bucket credentials` CLI outputs `AWS_S3_BUCKET_NAME` (not `BUCKET`) in the default format, and `BUCKET` is the variable name in Railway's variable reference system. The existing FilesService reads `STORAGE_BUCKET_NAME`. When wiring Railway variable references, map `${{bucket.BUCKET}}` → `STORAGE_BUCKET_NAME` (or rename the env var in FilesService to match Railway's output directly).

### 5. `AWS_ENDPOINT_URL` auto-pickup by SDK

AWS SDK v3 reads `AWS_ENDPOINT_URL` from the environment automatically (no need to pass `endpoint` to the constructor if this env var is set). However, the existing FilesService explicitly passes `endpoint: process.env.AWS_ENDPOINT_URL` to the constructor — this is fine and makes the code explicit. Do not double-set it (constructor option takes precedence over env var when both are present).

### 6. Railway Buckets public URL pattern for avatar serving

The existing `buildPublicUrl` function in FilesService builds URLs as `<endpoint>/<bucket>/<key>` (path-style). For virtual-hosted buckets, the correct public URL pattern is `https://<bucket-name-hash>.storage.railway.app/<key>`. Railway Buckets do not support public access — all file serving must go through presigned GET URLs or a backend proxy. The existing `resolvePublicUrl` that builds a "public URL" for avatar persistence is incorrect for Railway Buckets (all buckets are private). Fix: store the `key` on the user record, generate a presigned GET URL at read time instead of persisting a static URL.

### 7. CORS required for direct browser uploads

For presigned-PUT uploads from a browser frontend, the bucket must have a CORS configuration that allows the frontend origin and the `PUT` method. Configure via `aws s3api put-bucket-cors` against the Railway bucket endpoint. Without CORS, the browser's preflight `OPTIONS` request is blocked. The Railway docs provide the exact CLI command.

### 8. SDK v3 version pinning — all @aws-sdk/* packages must be at the same minor version

`@aws-sdk/client-s3` and `@aws-sdk/s3-request-presigner` must be at the same version. Mixed versions (e.g., client-s3 at 3.1075.0, presigner at 3.1000.0) cause `@smithy` peer dependency conflicts. The existing package.json pins both at `^3.1075.0` — keep them in sync on every upgrade.

## Integration-Specific Findings

### Our adapter patterns

The existing `FilesService` at `apps/api/src/files/files.service.ts` is the canonical integration point. It uses:
- Lazy S3Client initialization (null-safe; returns 503 when env unconfigured)
- `PutObjectCommand` + `getSignedUrl` for presigned-PUT uploads
- `HeadObjectCommand` for server-side size enforcement at confirm time
- `buildPublicUrl` helper for URL construction (see Gotcha #6 — needs review for wave-19)

For wave-19 attachment uploads (M3), extend `FilesService` with:
- `presignAttachmentUpload(userId, channelId, messageId, contentType, filename)` — key schema: `attachments/<channelId>/<messageId>/<uuid>.<ext>`
- `checkAttachmentSize(key)` — HeadObject; throw 413 if > 10MB
- Content-type allowlist for attachments (images + common file types: PDF, DOCX, XLSX, TXT, ZIP — confirm exact allowlist at P-2 Spec)
- `generateAttachmentDownloadUrl(key)` — presigned GET, 1-hour TTL

### Env var configuration on Railway

Railway bucket credentials are injected as env vars. The env var names the FilesService currently reads:

| FilesService reads | Railway Bucket provides (variable reference) | Notes |
|---|---|---|
| `AWS_ENDPOINT_URL` | `${{bucket.ENDPOINT}}` | Required; full URL e.g. `https://storage.railway.app` |
| `AWS_ACCESS_KEY_ID` | `${{bucket.ACCESS_KEY_ID}}` | Required |
| `AWS_SECRET_ACCESS_KEY` | `${{bucket.SECRET_ACCESS_KEY}}` | Required |
| `AWS_REGION` | `${{bucket.REGION}}` | Value: `auto` for Railway/Tigris |
| `STORAGE_BUCKET_NAME` | `${{bucket.BUCKET}}` | The actual S3 bucket name (includes hash suffix) |

The Railway CLI's `railway bucket credentials` command outputs `AWS_*` prefixed names matching what the AWS SDK reads by default. The auto-inject feature in Railway dashboard (AWS SDK preset) sets all vars automatically.

**No self-generated secrets.** Railway Buckets credentials are account-issued by Railway's own infrastructure — not self-generated via `openssl`. The deployer copies them from the Railway dashboard. This is the "account-issued credential" pattern in always-on rule 6, but the account in question is the existing Railway project (not a new external account), so no founder cred-ask for a new service account is required.

### Bugs we hit and how we solved them

None yet — wave-19 is the first attachment wave. Avatar uploads via FilesService have been in production since wave-N (see waves before 18).

### What differed from the official docs

- Railway Buckets URL style change: older buckets need `forcePathStyle: true`; newer buckets use virtual-hosted style. The `forcePathStyle: true` in FilesService was correct for the existing bucket at creation time. Wave-19 bucket provisioning must check the Credentials tab and set the flag accordingly.
- Railway Buckets endpoint is `https://storage.railway.app` (not a Tigris URL directly). The Tigris URL (`fly.storage.tigris.dev`) referenced in the architecture/sdks.md was the previous endpoint format; Railway now proxies through `storage.railway.app`.
- All S3 API operations on Railway Buckets are free and unlimited — no per-request cost even for presigned URL operations. This differs from AWS S3 (Class A/B operation pricing).
