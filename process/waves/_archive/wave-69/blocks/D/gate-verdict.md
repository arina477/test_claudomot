## Rationale

The one surviving defect from attempt 1 — the `showToast()` factory emitting bare `<div>` toasts with no `role`/`aria-live`, silencing the inbox-action network-error state to screen readers (WCAG 4.1.3 failure, and the toast is the *only* feedback channel for that error) — is genuinely and correctly fixed. I read `design/staging/moderation-report.html` myself rather than trusting the reviewer summaries: at lines 807–808, `showToast()` calls `toast.setAttribute('role', type === 'error' ? 'alert' : 'status')` then `toast.setAttribute('aria-live', type === 'error' ? 'assertive' : 'polite')`. Both calls execute after `document.createElement('div')` (line 805) and *before* `className` (816), `innerHTML` (817), and `container.appendChild(toast)` (819) — so the live-region contract is stamped on the element before it enters the DOM, which is the required announcement pattern. The message text is a `<span class="font-medium">${message}</span>` descendant of that same role-bearing `toast` div (817), so the live-region announcement fires over the message subtree. Error toasts get `role="alert"` + `aria-live="assertive"` (interrupts — correct for an action failure demanding attention); success/info get `role="status"` + `aria-live="polite"` (queues — correct for confirmations). Both error call-sites pass `type='error'` (submitReport line 728, handleAction line 764) and both success call-sites pass `type='success'` (lines 738, 782), so every feedback path — including the previously-silent inbox-action failure — is now announced. WCAG 4.1.3 clears.

The remedy was ARIA-only (~2 lines in one function): no visual, token, layout, copy, or interaction change, so there was no surface for regression — and I independently re-confirmed every attempt-1-approved item still holds against the live file: danger buttons `--danger-btn` #b91c1c on white = 6.47:1 (AA pass, hover #991b1b = 8.31:1); emerald primary `--surface-950` #0a0a0b dark text on #10b981 = 7.80:1; nav badge on `--danger-btn` (line 238), not raw `--danger`; informational 11px text on `--text-secondary` (~5.97:1) with `--text-muted` confined to placeholders/decorative; modal `role="dialog"` + `aria-modal` + `aria-labelledby` with a real Tab/Shift+Tab focus-trap and Esc-close (lines 514, 621–643); loading skeletons (not spinners) per §8; mobile bottom sheet (items-end → sm:items-center, rounded-t on <640px); viewport meta with no `maximum-scale` (1.4.4 pass); report affordances on all three target types — server discovery card, member row, message hover bar — each `aria-label`led; Phosphor icons only; full token discipline traced to DESIGN-SYSTEM §1–§8; brief §9 success criteria all met. Both post-rework dual reviewers returned APPROVE (Phase-1 matrix APPROVE/APPROVE), and their independent findings match my own read of the file. The single `moderation-report` gap clears the bar; no gap falls short. The non-blocking implementation-level notes the reviewers carried forward — inline `bg-[#10b981]` at line 392 (the exact `--accent-emerald` value, a token-hygiene note, not an invented hex or fragmentation); `title`-only React/Reply demo buttons in the affordances rail (outside brief scope); and optional aria-describedby / aria-labelledby / focus-restoration-on-close / prefers-reduced-motion refinements — are not design failures and do not gate adoption; they are handoff notes for B-block.
