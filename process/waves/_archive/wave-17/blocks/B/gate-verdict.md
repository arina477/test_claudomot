## Rationale

This wave ships a single test-infra deliverable: a real-Postgres integration test that proves `createServer`'s atomic transaction genuinely rolls back on a mid-transaction failure — closing the honesty gap left by the unit test, which mocked `db.transaction` and therefore proved nothing about rollback. The gate lens here is load-bearing because a rollback test that *looks* like it proves rollback but secretly mocks the transaction away is worse than no test. I scrutinized that specifically and the proof is genuine, not coverage-theater. The mid-txn rollback case captures the REAL `db.transaction` (`realTransaction = dbModule.db.transaction.bind(dbModule.db)`) BEFORE spying, then its `mockImplementation` invokes `realTransaction(...)` — so the real drizzle transaction runs against real Postgres. The spy's ONLY fault injection is wrapping `tx.insert` to count calls and throw on the 5th (channels), while inserts 1–4 (server, role, member, category) execute through the unmodified `originalInsert` inside the open real transaction. The throw propagates → drizzle issues a real ROLLBACK → no-orphan-rows is asserted via REAL row counts (`countRows` runs over the separate harness pool against the live test DB), and all five tables read 0. Because the harness pool is a distinct connection, reading 0 rows is a true cross-connection commit-visibility check that can only pass if the SUT's own connection actually rolled back. The first-insert case uses a genuinely real constraint: `servers.invite_code` carries `.unique()` (verified in schema), the pre-seeded collision yields a real Postgres 23505, and rollback leaves only the pre-seeded row. The positive case commits all five row-kinds (counts = 1 each), proving the transaction is not a no-op and migrations applied. CF-2 holds: `pg-harness` is the first import and sets `process.env.DATABASE_URL = DATABASE_URL_TEST` at module-eval time before the lazy Proxy in `src/db/index.ts` resolves its Pool; the SUT imports that same `db` singleton, and the spy targets that exact singleton — `createServer`'s real db hits the test DB, NOT a side instance. The harness pool is correctly separate and used only for truncate/fixture/count. The integration vitest PROJECT (not just the one spec) is parallel-safe: `fileParallelism: false` + `pool: 'forks'` + `singleFork: true` at the project level, so the reusable harness (consumer 02fa8011) inherits serial execution and the shared-DB truncate-between-cases strategy stays sound for any number of integration specs. `migrate` is fail-loud (no try/catch) and both 0004_* migrations are journaled (idx 3 + 4); the harness applies all of them. `test:ci` runs the 292 unit tests first (`vitest run`) then integration, so unit coverage is preserved; `describe.skipIf(!DATABASE_URL_TEST)` plus an explicit `it.skip` message gives CI-runs / local-skips-with-clear-reason (not a silent pass). No product code, schema, or dependency changed; boot-safety is n/a for test-only code; no scale gold-plating. The structure is CI-correct and will genuinely prove rollback when CI runs it — C-1 CI is the authoritative green. APPROVED for Phase 2 (/review).
