- **Transactional reply_count realistic against existing txn pattern — PARTIALLY WRONG (non-blocking).** P-3 line 8 + the prompt's framing assert "createServer/createMessage use `db.transaction`." Grep: `db.transaction` exists ONLY in `servers.service.ts:68,440` + `rbac/owner-lockout.service.ts`. **`createMessage` does NOT use `db.transaction`** — it is a sequence of standalone `await db.insert/select` calls (`messages.service.ts:255-317`), no `tx` wrapper. So the same-txn "insert reply + UPDATE parent count atomically" requirement is a NEW transactional boundary the build must introduce; it is NOT inherited from createMessage. **This does NOT sink the plan** — `db.transaction(async (tx) => …)` is a proven, in-repo pattern (servers.service `:68` wraps multi-table inserts exactly this way), so B-3 has a real template. But the plan's wording overstates reuse; the build must wrap createReply in `db.transaction`, not copy createMessage's (non-transactional) shape. Logged for B-3 awareness; recommend the B-3 prompt point the implementer at `servers.service.ts:68` (not createMessage) as the txn template.
